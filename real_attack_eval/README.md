# Real-Attack Evaluation

Victim-side syscall traces from live attack runs, evaluated with the trained Phase 5 fusion detector. This folder is self-contained: it carries the inference code, selected model artifacts, and the shared ADFA-LD benign window pool used as the negative comparison set.

The current Phase 5 model is the XGBoost meta-classifier from the 32-bit pipeline. It combines three scores:

- Path A: TF-IDF + PCA + SGD one-class SVM
- Path B: CNN-1D autoencoder
- Path C: GRU next-syscall predictor

The model manifest reports `AUC_test = 0.9113` on the original ADFA-LD held-out protocol. The operating threshold used here is `tau* = 0.401214`.

## Quick Start

Install the runtime dependencies:

```bash
pip install numpy scikit-learn xgboost torch
```

Run all families:

```bash
bash real_attack_eval/run_all.sh
```

Run one family:

```bash
bash real_attack_eval/meterpreter/scripts/run_eval.sh
```

Run only the novel families:

```bash
bash real_attack_eval/run_all.sh reverse_shell sqlmap nikto shellshock cryptominer
```

For standalone use, set `CAPTURE_ROOT` to the parent directory containing the capture folders:

```bash
export CAPTURE_ROOT=/path/to/captures
bash real_attack_eval/run_all.sh
```

Expected capture path shape:

```text
<CAPTURE_ROOT>/<family>_capture/adfa_ld_out/Attack_Data_Master/<Family>_Real/
```

## Evaluation Protocol

All current results use the same protocol:

1. Capture victim-side syscall traces with `strace -f`.
2. Convert traces to ADFA-LD-style i386 syscall-number sequences.
3. Build `artifacts/manifest.json`.
4. Score sliding windows with `scripts/realtime_eval/export_window_scores.py`.
5. Compute metrics with each family `compute_metrics.py`.

Parameters:

- window size: `20`
- stride: `2`
- syscall ABI: `i386`
- benign pool: `50,000` ADFA-LD benign windows
- benign test split: `40%` by trace group
- seed: `42`
- threshold: `tau* = 0.401214`

Reported metrics:

- `Trace-DR`: percent of traces with at least one flagged window
- `Window-DR`: percent of attack windows above `tau*`
- `AUC`: attack windows vs sampled ADFA-LD benign test windows
- `AUCPR(1:1)`: average precision after equal positive/negative sampling
- `TPR@FPR1%` and `TPR@FPR0.1%`: low-false-positive operating points

Raw AUCPR and F1 are stored in the JSON artifacts, but they depend strongly on class ratio and the ADFA-LD-calibrated threshold.

These families are not part of ADFA-LD training. They do not have paper/LOAO baseline constants.

| Family | Why it is novel here | Expected syscall signal | AUC | Window-DR |
|---|---|---|---:|---:|
| Reverse Shell | Bash-native socket to shell control | socket/exec/read/write bursts | 0.7289 | 66.2% |
| SQLMap | SQL injection tool, not an ADFA-LD family | repeated HTTP and SQLite I/O | 0.8900 | 98.6% |
| Nikto | Web vulnerability scanner | dense request/response loops | 0.8608 | 99.2% |
| Shellshock | CGI/bash CVE workflow | short bash execution phase | 0.6227 | 29.5% |
| Cryptominer | CPU resource-abuse simulation | thread and CPU-loop behavior | 0.7040 | 55.3% |

SQLMap and Nikto transfer best because the attack signal appears in most windows. Shellshock is sparse: all traces are flagged at least once, but most windows remain below threshold.

## How To Read The Results

Trace-DR is a presence metric. A trace is counted as detected if one or more windows cross `tau*`. Long traces have many chances to cross the threshold, so Trace-DR can look strong even when the window ranking is weak.

Window-DR, AUC, AUCPR(1:1), and low-FPR TPR are stricter. They show that the model often catches attack presence, but the ADFA-LD threshold is not stable enough for low-alert operation on every real family.

The fair summary is:

> The Phase 5 fusion detector flags almost all real attack traces at least once, including novel families. The weaker point is window-level calibration and low-FPR behavior under environment shift.

Avoid writing this as a blanket "100% real-world detection" claim without the trace/window distinction.

## Capture Infrastructure

Pre-captured traces for all ten families are published on Zenodo:

> **DOI: [10.5281/zenodo.20776518](https://doi.org/10.5281/zenodo.20776518)**

Download and extract each `<family>_capture.zip` archive under `datasets/` to use the pre-built traces directly. Each archive includes `raw_strace/`, `adfa_ld_out/`, and `docker/` for re-capture.

| Web Shell | Apache + DVWA + PHP | apache2 workers | 8 |
| Meterpreter | vsftpd 2.3.4 CVE-2011-2523 | vsftpd backdoor shell | 10 |
| Java Meterpreter | i386/debian + JRE + MSF | Java JVM with `strace -f` | 8 |
| Reverse Shell | i386/debian + Python bind shell | Python + bash children | 6 |
| SQLMap | i386/debian + Apache + PHP/SQLite | apache2 workers | 5 |
| Nikto | i386/debian + Apache | apache2 workers | 4 |
| Shellshock | i386/debian + bash 4.3.0 + Apache CGI | apache2 + bash children | 6 |

To re-run captures from scratch (authorized lab environment only), all scripts support incremental runs with `[n_runs] [start_id]`:


## Adding A New Attack Family

1. Create capture infrastructure in `datasets/<name>_capture/`.
2. Use `SYS_PTRACE` and `seccomp:unconfined` on the victim container.
3. Produce ADFA-LD-style syscall-number traces.
4. Add `real_attack_eval/<name>/scripts/build_manifest.py`.
5. Add `real_attack_eval/<name>/scripts/compute_metrics.py`.
6. Add `real_attack_eval/<name>/scripts/run_eval.sh`.
7. Add the family name to `real_attack_eval/run_all.sh`.
8. Run the capture script, then run the evaluation script.
